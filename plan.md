# Word Golf — Plan

**Working title:** Word Golf (name TBD — see Open Questions)
**One-liner:** A tiny, replayable word puzzle that doubles as a live showcase of the LaunchDarkly AutoFactory — autonomous, metric-guarded releases with automatic rollback — and as a clonable sample app for a "build your own software factory" workshop.

---

## 1. Why this app

This single codebase has to serve three audiences at once:

1. **The public / a website visitor** — plays a fun game, and watches a "mission control" panel where AI agents ship guarded releases to the very game they're playing and roll back the ones that hurt a metric. No human in the loop.
2. **LaunchDarkly execs (the pitch)** — must "get it" in under 30 seconds. The game's core mechanic *is* the rollback story: change a letter, and if it isn't a real word, you revert to the last good state. Game concept and platform concept are the same shape.
3. **Workshop attendees** — clone the repo, run one bootstrap command to wire the app into *their own* LaunchDarkly project, open a PR, and watch the AutoFactory flag it, instrument a metric, run a guarded release, and auto-rollback on regression. The game is just the concrete, fun payload; the lesson is the LaunchDarkly loop.

The design principle that resolves every trade-off: **keep the game small and self-explanatory so the LaunchDarkly concepts are always the star.**

## 2. The game

### 2.1 Rules

- A puzzle is a **start word** and a **target word**, both valid 5-letter words.
- Each move: change **exactly one letter** to form another **valid** 5-letter word (must exist in the allowed-words list).
- Reach the target in as few moves as possible.
- **Par** = the shortest possible number of moves (computed by BFS over the word graph). Score relative to par: eagle (−2), birdie (−1), par (0), bogey (+1), …
- An **invalid move** (not a real word, or more than one letter changed) is rejected and the board stays on the last valid word. This is the rollback metaphor — surface it as such in copy.
- **Undo** reverts to the previous word (another "rollback" beat).

### 2.2 Why it can't get stuck

Daily/endless targets are generated by a **random walk** from the start word along valid neighbors, so a solvable path always exists. Par is then computed independently by BFS (it may be shorter than the walk length).

### 2.3 Modes (these drive replay + data)

| Mode | Description | Primary purpose |
| --- | --- | --- |
| **Daily** | One seeded puzzle per day, identical for everyone. Streaks + shareable result. | Habit loop (the Wordle effect), retention data |
| **Endless / Practice** | Infinite random puzzles, "beat par." | Volume of plays + metric data |
| **Speedrun** (optional, later) | Timed; fewest seconds to solve. | Latency-metric data, competitive replay |

### 2.4 Shareable result (growth loop)

Spoiler-free text share, e.g.:
`Word Golf #128 — solved in 5 (par 4) 🏌️ +1`
Include a small per-move emoji track for flavor. No words revealed.

## 3. How it maps to the AutoFactory

Grounded in `launchdarkly-labs/launchdarkly-auto-factory`: agents (defined in AgentControl) classify a PR, create a flag (off), instrument **error / latency / business** guarded-release metrics, write flag-on/flag-off tests, and post a verdict; then Beacon turns the flag on as a **guarded release** and monitors it to a terminal state — **completed** or **reverted by a guardrail metric**.

### 3.1 The three guarded-release metrics, mapped to the game

| AutoFactory metric | Word Golf event | Example use |
| --- | --- | --- |
| **Business** | `puzzle_completed`, `made_par`, `daily_returned`, `result_shared` | Did the change help people finish / come back? |
| **Latency** | `time_to_solve_ms`, `move_latency_ms` | Did the change slow play down? |
| **Error** | `invalid_move`, `puzzle_abandoned` (give-up / rage-quit) | Did the change frustrate people? |

### 3.2 Flag-worthy features (what agents tune / what attendees flag)

Each is a clean, single-hypothesis A/B target — small enough to hold in your head, which is exactly what an exec demo and a workshop need.

| Flag key | Type | Hypothesis | Watched metric |
| --- | --- | --- | --- |
| `hint-button` | bool | A hint raises completion… but may hurt "made par." | business: `made_par` |
| `par-algorithm` | string (`shortest` / `no-reuse` / `heuristic`) | Different par math changes perceived difficulty. | business: `puzzle_completed` |
| `word-pool-difficulty` | string (`easy`/`medium`/`hard`) | Harder pairs lower completion but raise engagement. | business + error |
| `daily-theme` | string | Cosmetic; null hypothesis (teaches a "no-impact → promote" run). | business: `daily_returned` |
| `show-mission-control` | bool | Operational toggle for the showcase panel. | n/a (kill switch) |

### 3.3 The "agents decide what's relevant" beat

When a feature PR arrives (e.g., the hint button), the agent chain should visibly reason: *"This adds assistance. The risk isn't completion — it's that we trivialize the challenge. Watch `made_par`, not `puzzle_completed`."* It then creates that metric, instruments the event, writes flag-on/off tests, and posts a verdict. This is the AgentControl "agents choosing the metric" story in one screen.

## 4. Mission Control panel (the showcase centerpiece)

A side panel (stacks below the game on mobile) that visualizes the factory loop as a live feed:

```
PR detected → [Classify] → [Create flag] → [Instrument metric] → [Write tests] → [Verdict]
            → Guarded release @ 20% ▸ watching `made_par` ▸ live sparkline
            → Terminal: ✅ Completed (promoted to 100%)  |  ↩︎ Reverted (guardrail tripped)
```

- **Player tie-in:** "You're in the 20% test cohort for *Hint Button*." On rollback: "Guardrail tripped on made-par rate — reverting you to the stable build." This makes rollback *felt*, not just shown.
- **Two data sources, clearly labeled:**
  - **Simulated** (default for the public page + the exec demo): a deterministic state machine that scripts believable scenarios. Reliable, no per-visitor cost, demo-safe.
  - **Live** (workshop / internal): reads real flag + guarded-release status from LaunchDarkly for the page's own experiments.
- Mode is explicit in the UI so we never misrepresent what's real.

## 5. Architecture

### 5.1 Stack

- **Vite + React + TypeScript** single-page app.
- **LaunchDarkly React Client SDK** for flag evaluation; `track()` for metric events.
- **Styling:** follow the LaunchDarkly design system (Söhne/Sora type, brand tokens, gradients) for anything public-facing or exec-facing. Apply the `launchdarkly-design-system` skill at build time.
- **Word data:** bundled static files — full allowed-guess list (~12,972 words) as graph nodes; answers list (~2,315) as the friendly pool for choosing daily start/target words.
- **Persistence:** `localStorage` for streaks/stats (fine in the real app; not an artifact constraint).
- **Backend:** none required for v1. Leaderboards/aggregates, if added, go through LaunchDarkly metrics or a tiny serverless function. Guarded releases are driven server-side by the AutoFactory/Beacon, not by the client.

### 5.2 Par / graph performance (do this once at load)

Build **wildcard buckets** for O(1)-ish neighbor lookup: for each word, generate its 5 one-position-wildcarded patterns (`_ATCH`, `C_TCH`, …) into a `Map<pattern, word[]>`. Neighbors of a word = union of its 5 bucket members minus itself. BFS over this is fast even at 13k nodes. Document this so workshop attendees understand it.

### 5.3 Daily puzzle generation

Seed an RNG by UTC date → pick start from the answers pool → random-walk `K` steps (K set by `word-pool-difficulty`) avoiding immediate backtracks → that's the target. Par = BFS(start, target). Everyone gets the same daily; results are comparable.

### 5.4 Repo layout (mirrors auto-factory conventions where useful)

```
word-golf/
  apps/web/                 # React game + Mission Control UI
  packages/engine/          # pure TS: graph, BFS par, move validation, scoring, daily seed
  packages/ld/              # LD client init, flag keys, typed metric-event helpers
  packages/mission-control/ # factory-loop visualization: simulated + live adapters
  data/                     # allowed-words.txt, answers.txt
  config/                   # declarative flag + metric definitions (source of truth)
  scripts/bootstrap/        # provisions flags + metrics into the attendee's LD project
  examples/feature-prs/     # prepared, flag-worthy changes for the workshop
  docs/                     # workshop guide, architecture, demo script
  .github/workflows/        # hook to launchdarkly-auto-factory (Phase 1 action)
  .env.example
  README.md
```

## 6. Workshop flow (the education tool)

The repo is intentionally a clean **demo-app** that the existing `launchdarkly-auto-factory` can operate on.

1. `git clone … && cd word-golf && npm install`
2. `cp .env.example .env` → fill `LD_SDK_KEY`, `LD_API_KEY`, `LD_PROJECT_KEY`, `ANTHROPIC_API_KEY`.
3. `npm run bootstrap` — idempotently provisions the flags + the three metrics into your LD project from `config/`. (Mirrors the auto-factory bootstrap.)
4. `npm run dev` — play locally; watch flags flip behavior and metric events fire.
5. Add the auto-factory GitHub Action workflow; open a PR from `examples/feature-prs/` (start with `hint-button`).
6. Watch the agent chain classify the PR, create the flag, instrument the metric, write tests, post a verdict.
7. Deploy; Beacon runs the guarded release → it **completes** or the guardrail **rolls it back**.
8. Mission Control shows the whole thing live.

**Time-box target:** clone → first guarded release on screen in **under 30 minutes**.

## 7. Milestones

Ship something playable fast, then layer LaunchDarkly, then the showcase, then the workshop packaging.

### Phase 0 — Playable core (no LaunchDarkly)
- Scaffold monorepo; load word lists; build wildcard-bucket graph.
- `engine`: BFS par, move validation, scoring, daily seed + random-walk generator.
- Minimal playable Daily mode UI (input, board, par, win state, undo).
- **Exit criteria:** a person can play and finish a daily puzzle.

### Phase 1 — Replayability + local metrics
- Endless/Practice mode; stats, streaks, shareable result text.
- Emit metric events to a local stub (`puzzle_completed`, `time_to_solve_ms`, `invalid_move`, `puzzle_abandoned`, `made_par`, `result_shared`, `daily_returned`).
- **Exit criteria:** all modes work; events fire with correct payloads.

### Phase 2 — LaunchDarkly integration (the "real" usage)
- Wire LD React SDK; gate `hint-button`, `par-algorithm`, `word-pool-difficulty`, `daily-theme`, `show-mission-control`.
- Route metric events to LD `track()`.
- `config/` declarative definitions + `scripts/bootstrap` to provision them.
- **Exit criteria:** flags change behavior live; metrics land in LD; bootstrap is idempotent.

### Phase 3 — Mission Control panel
- Simulated factory loop first (deterministic, demo-safe) — this is the exec-demo build.
- Player-cohort tie-in + rollback messaging.
- Live adapter that reads real flag/guarded-release status.
- **Exit criteria:** a 30-second demo runs reliably end-to-end in Simulated mode.

### Phase 4 — Workshop packaging
- `examples/feature-prs/` (hint button, harder pool, par-algorithm swap) ready to open.
- `.github/workflows/` hook to launchdarkly-auto-factory.
- `docs/`: workshop guide, architecture, demo script.
- **Exit criteria:** a clean clone reaches a guarded release on screen in <30 min.

### Phase 5 — Polish + public launch
- Apply LaunchDarkly brand/design system; mobile layout pass; accessibility.
- Deploy public; finalize name.
- **Exit criteria:** public page live; exec pitch rehearsed.

## 8. The 30-second exec demo script

1. "This is Word Golf — turn one word into another, one letter at a time. Watch: I type a non-word… it bounces back to the last good state. That's a rollback." (5s)
2. "Now look at the panel. An agent just shipped a *Hint Button* to 20% of players and decided the metric to watch is 'made par' — because a hint could help people finish but cheapen the win." (10s)
3. "Made-par rate is dropping for the test group… guardrail trips… and it rolls itself back. No human touched it." (10s)
4. "Same codebase is what we hand workshop attendees — they clone it and wire this exact loop into their own LaunchDarkly project." (5s)

## 10. Risks & mitigations

| Risk | Mitigation |
| --- | --- |
| BFS too slow on full list | Precompute wildcard buckets at load; cache par per puzzle. |
| Unsolvable start/target pairs | Generate target by random walk → path guaranteed. |
| Live demo flakiness in front of execs | Default Mission Control to deterministic Simulated mode. |
| Per-visitor agent cost on public page | No real agents per visitor; public loop is simulated. |
| Workshop runs over time | One-command idempotent bootstrap; pre-baked feature PRs. |
| Game complexity creeping up and burying the LD point | Hard rule: every feature must map to a flag + a metric, or it's cut. |
